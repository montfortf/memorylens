The Problem

Memory debugging is impossible today

AI agents fail in subtle, hard-to-reproduce ways. Your agent "forgot" a critical fact. Retrieval returned the wrong memories. Compression destroyed context. But you have no idea when it happened or why.

Every major observability tool — LangSmith, Langfuse, Phoenix — traces LLM calls. None of them instrument the memory pipeline. The writes, reads, compressions, and updates that determine what your agent knows are completely invisible.

MemoryLens closes that gap. Instrument once. See everything.

🔮
The silent drop
A memory write returns success but the backend silently drops the content due to a deduplication policy. Your agent never learns this happened.
🔍
The wrong retrieval
Search returns results with 0.31 similarity scores when your threshold is 0.4. The agent gets no memories — but the error is invisible upstream.
📉
Lossy compression
Summarization compresses 5,000 tokens to 300. The semantic loss score is 0.78. Key facts are gone. There's no audit trail.
💸
Unknown memory costs
Embedding generation, LLM compression, vector store operations — what's the real cost per session? Without attribution, you can't optimize.
Capabilities

Full visibility across the memory pipeline

Six instruments covering every stage — from the moment content is written to when it's retrieved, compressed, or lost.

🔬
01 — Write-Path Tracing
See every memory write, including silent drops

Instrument memory.write and memory.update spans across any backend. Capture the content stored, embedding model used, vector dimensions, and drop policies triggered. Know exactly when and why a write succeeded — or was silently discarded.

@instrument_write
🎯
02 — Retrieval Debugger
Debug why retrieval returned wrong results

Every memory.read span records the query, results count, per-result similarity scores, threshold, and top-k. Visualize the score distribution. Find the retrieval misses and threshold miscalibrations that make your agent hallucinate.

@instrument_read
📊
03 — Compression Auditor
Measure semantic loss in memory summarization

Track memory.compress spans with pre/post content, compression ratio, semantic loss score, and the model used. Detect when summarization is destroying critical context before it affects agent quality. Compare compression strategies head-to-head.

@instrument_compress
📡
04 — Memory Drift Detection
Track how memory evolves across sessions

Instrument memory.update spans with previous and new content versions, update type (merge/replace/append), and memory key lineage. Build a timeline of how agent knowledge changes — and catch regressions where updates overwrite correct information.

@instrument_update
💰
05 — Cost Attribution
Know the real cost of your memory pipeline

Attribute embedding generation, LLM compression, and vector store operations to agents, sessions, and users. Break down costs by operation type. Identify the expensive sessions and optimize where it matters most. Export as JSONL for billing integrations.

memorylens stats
🔔
06 — Alerting
Get notified when memory health degrades

Set thresholds on error rates, drop rates, retrieval scores, and compression ratios. Route alerts to any OTLP-compatible backend — Grafana, Datadog, New Relic, Honeycomb. Self-hosted means your memory content never leaves your infrastructure.

otlp export
How It Works

Instrument → Trace → Debug

1
Instrument your memory functions

Decorate existing memory functions with @instrument_write, @instrument_read, and friends — or use auto-instrumentation for LangChain and Mem0 with zero code changes. Call memorylens.init() once at startup.

2
Traces flow to your backend

Every memory operation generates a structured span — operation type, timing, content, scores, status. Spans are flushed asynchronously via BatchSpanProcessor with <2ms p99 overhead. Local SQLite by default, OTLP for production.

3
Debug with CLI or dashboard

Use memorylens traces list to browse, filter by operation/status/time, and tail live. Use memorylens traces show <id> for full span detail. Or ship to Grafana, Honeycomb, or any OTLP backend for team dashboards.

Python 
import memorylens

# 1. Initialize once
memorylens.init()

# 2. Decorate your memory functions
@memorylens.instrument_write(backend="my_db")
def store(content: str) -> bool:
    return db.save(content)

@memorylens.instrument_read(backend="my_db")
def search(query: str) -> list[str]:
    return db.search(query)

# 3. Add session context
with memorylens.context(
    agent_id="support-bot",
    session_id="sess-123",
    user_id="user-456"
):
    store("user prefers vegetarian meals")
    results = search("dietary preferences")

# Or: zero-code auto-instrumentation
memorylens.init(instrument=["langchain", "mem0"])

Quick Start

Up and running in minutes

Instrument your first memory function with two lines of code. Everything is captured locally by default — no external service required.

Python
bash 
pip install memorylens
# With LangChain or Mem0 support:
pip install memorylens[langchain]
pip install memorylens[mem0]
Python 
import memorylens

# Initialize — SQLite local storage by default
memorylens.init()

# Manual instrumentation
@memorylens.instrument_write(backend="mydb")
def store_memory(content: str) -> bool:
    return db.save(content)

@memorylens.instrument_read(backend="mydb")
def search_memory(query: str) -> list[str]:
    return db.search(query)

# Auto-instrumentation for supported frameworks
memorylens.init(instrument=["langchain", "mem0"])

# Inspect traces in the CLI
# $ memorylens traces list
# $ memorylens traces show <trace-id>
# $ memorylens stats
TypeScript
bash 
npm install memorylens
# or
pnpm add memorylens
TypeScript 
import memorylens from 'memorylens'

// Initialize
memorylens.init()

// Wrap memory functions
const storeMemory = memorylens.instrumentWrite(
  async (content: string) => {
    return await db.save(content)
  },
  { backend: 'mydb' }
)

const searchMemory = memorylens.instrumentRead(
  async (query: string) => {
    return await db.search(query)
  },
  { backend: 'mydb' }
)

// Session context
await memorylens.withContext(
  { agentId: 'support-bot', sessionId: 'sess-123' },
  async () => {
    await storeMemory('user prefers vegetarian meals')
    const results = await searchMemory('dietary preferences')
  }
)
CLI Commands
bash 
memorylens init                               # Set up local storage
memorylens traces list                        # Browse all traces
memorylens traces list --status error         # Filter by status
memorylens traces list --operation read       # Filter by operation
memorylens traces list --last 1h              # Last hour only
memorylens traces show <trace-id>            # Inspect a trace
memorylens traces tail                        # Live tail
memorylens traces export -o traces.jsonl      # Export as JSONL
memorylens stats                              # Summary statistics
memorylens stats --group-by operation         # Breakdown by type

🔭
OpenTelemetry Native

MemorySpan maps 1:1 to OTel spans. Every attribute — operation, backend, scores, content — is a proper OTel attribute. Send to any OTLP-compatible backend with standard env vars. No vendor lock-in.

🏠
Self-Hosted by Default

Traces stay in ~/.memorylens/traces.db on your machine by default. Your agent's memory content — which may include PII — never leaves your infrastructure unless you explicitly configure an OTLP endpoint.

⚡
Non-Blocking, <2ms Overhead

The BatchSpanProcessor enqueues spans to a background thread. The instrumented function returns immediately. p99 overhead is under 2ms — safe for production memory pipelines with tight latency budgets.

🔓
Open Source, Apache 2.0

Inspect the code, extend it, contribute back. The instrumentation protocol spec is public — build integrations for any memory framework. Validate community instrumentors with memorylens validate integration.
